# RWA 旗舰项目 · 全栈开发计划(EVM,面试主菜)

> 工作名:**`rwa-credit`**(代币化 RWA + 抵押借贷;你可改名)。三吃:面试作品集主菜 / RWA 域知识(你的优先方向)/ 证明 TRON 全栈能力可迁移到 EVM 原生栈。
> 一句话定位:**「合规的代币化国债,且能拿去借出稳定币而不赎回」**——RWA(代币化+合规+NAV)× 你最强的借贷(隔离市场),合体。

---

## 1. 为什么是这个项目(面试视角)
- **RWA 是 2026 增量主线**,面试高频考点:合规转让限制、oracle/NAV、赎回与挤兑、权限模型、跨链。这个项目一次覆盖全部。
- **复用你最强的一块**:隔离借贷引擎直接搬 `lend-solvent` 的 vendored Morpho Blue,不重造轮子,把精力压在 **RWA 合规层 + NAV oracle + 许可型抵押的清算难题**这些**差异化**的地方。
- **可迁移性证明**:同一套全栈能力,链交互从 TronWeb 换成 **viem/wagmi**,是接口替换不是重写——正是块 A 对比文里的论点的实证。

## 2. 系统架构(四层)
```
┌─ 前端 (React+Vite+TS, wagmi/viem, RainbowKit) ─────────────┐
│  KYC 门控 · 申购/赎回 · 组合+收益(NAV) · 抵押借出 · 健康/清算  │
├─ 索引层 (ponder 或 viem 自建 + Postgres) ──────────────────┤
│  申购/赎回历史 · NAV 曲线 · 持仓与借贷头寸 · 清算事件          │
├─ 合约层 (Foundry / Solidity) ─────────────────────────────┤
│  ① 合规核心: IdentityRegistry(KYC 白名单)+ 转让限制 hook    │
│  ② RWA 代币: 许可型 ERC-20(申购 mint / 赎回 burn)          │
│  ③ NAV oracle: 链下资产估值上链(price-per-token + 收益累计)  │
│  ④ 申购/赎回: USDC ↔ RWA @NAV + 赎回队列(T+N)               │
│  ⑤ 隔离借贷市场: 复用 Morpho 引擎 + NAV-oracle 适配器          │
│     (RWA 当抵押 / USDC 当贷款,借出而不赎回)                  │
└────────────────────────────────────────────────────────────┘
```

## 3. 技术选型(EVM 原生,刻意区别于 TRON 栈)
| 层 | 选型 | 为什么 |
|---|---|---|
| 合约 | **Foundry**(forge/anvil) | 行业标准;你 lend-solvent 已用 |
| 借贷引擎 | **vendored Morpho Blue**(从 lend-solvent 搬) | 不重造;把精力放 RWA 差异化 |
| 合规 | **简化但忠实的 ERC-3643 子集**(IdentityRegistry + 转让 hook + agent 角色) | 全 T-REX 太重;子集足够展示理解,README 引用完整标准 |
| 前端 | **React+Vite+TS + wagmi/viem + RainbowKit** | EVM 原生钱包栈,对位你 TRON 的 tronwallet-adapter |
| 索引 | **ponder**(现代 EVM indexer)或 viem 自建 + Postgres | 对位你 lend-solvent 的 Go indexer |
| 测试网 | **Base Sepolia**(或 Sepolia) | L2、便宜、RWA 叙事常落 Base |
| Monorepo | pnpm + turbo,packages: config/core/sdk | 沿用 lend-solvent 结构 |

## 4. 里程碑(M0–M5,对齐 lend-solvent 节奏)

**M0 · 脚手架**:pnpm/turbo monorepo + Foundry + packages(config/core/sdk)+ CI。vendored Morpho 搬入。

**M1 · 合规核心(RWA 的灵魂)**
- `IdentityRegistry`:KYC 白名单(地址→已验证 + 可选 claim topics 简化版)。
- 许可型 RWA `ERC-20`:`_update`/transfer hook 里查 registry,非白名单 revert;agent 角色可 `mint/burn/freeze/forceTransfer/recover`(监管要求)。
- **诚实讨论点写进 README**:这些 agent 权限 = 中心化,但 RWA 合规**必须**有;取舍与缓解(多签/timelock/事件可审计)。
- forge 测:转让限制、freeze、forceTransfer、未 KYC 拒绝。

**M2 · NAV oracle + 申购/赎回**
- `NavOracle`:推 price-per-token(链下国债估值上链),收益靠 NAV 增长累计;**staleness 检查 + 熔断**(过期/异常价拒绝)。
- 申购:KYC 用户存 USDC → 按当前 NAV mint RWA。赎回:burn RWA → 按 NAV 收 USDC,**赎回队列(T+N 结算)**(RWA 真实约束)。
- forge 测:NAV 增长后申赎价正确、过期价拒绝、队列结算。

**M3 · 隔离借贷市场(复用 Morpho + RWA 难题)**
- 搬 Morpho 引擎,写 **NAV-oracle 适配器**(把 NavOracle 包成 Morpho 的 1e36-scale IOracle)。
- 建 RWA(抵押)/ USDC(贷款)隔离市场:supply / borrow / 健康 / 清算。
- **⭐ 差异化设计题(面试金句):许可型抵押的清算**——清算人 seize 到的是许可型 RWA,他**也得是 KYC 白名单**才能收。三种解法写进 README 对比:① 清算人白名单;② seize→强制赎回成 USDC;③ 协议托管再处置。选一个实现,讲清取舍。
- forge 测:NAV 下跌触发清算、KYC 清算人路径、坏账。

**M4 · 全栈前端 + 索引**
- wagmi/viem 前端:KYC 状态门、申购/赎回、组合+收益曲线、抵押借出、健康/清算面板。复用 lend-solvent 的**交易生命周期 SDK**思路(状态机/错误码)移植到 viem。
- 索引(ponder/viem):申赎历史、NAV 曲线、头寸、清算事件 → Postgres → 前端图表。

**M5 · 部署 + Demo + 架构文档**
- 部署 Base Sepolia + seed(mock USDC、几个 KYC 地址、初始 NAV)。
- Demo 脚本 + README + **架构决策文档**(合规模型选型、NAV oracle 信任模型、许可抵押清算三方案对比)——这份文档本身就是面试材料。

## 6. RWA 合规要点(面试必背,写进 README 显深度)
- **标准谱系**:ERC-3643(T-REX,带 IdentityRegistry + 模块化 compliance)vs ERC-1404(轻量 `detectTransferRestriction`)vs ERC-1400(分区/文档)。选 3643 子集,讲清为什么。
- **身份**:IdentityRegistry / OnchainID / claim topics(KYC、accredited、jurisdiction)。
- **转让限制**:`canTransfer` 模块化合规——白名单、锁定期、限额、jurisdiction 规则。
- **agent/issuer 权力**:mint/burn/freeze/forceTransfer/recovery——**为什么监管要求**(被盗找回、法院冻结)+ 中心化取舍如何缓解。
- **NAV/oracle 信任**:链下估值谁签、staleness、熔断;RWA 的"预言机=资产真值",比 DeFi 价格 oracle 责任更重。
- **赎回机制**:T+N 结算、队列、gating、挤兑场景。
- **许可抵押清算**:上面 M3 的三方案(这是 RWA×DeFi 最尖的设计题)。
- **监管框架背景**(点到为止显域感):Reg D/S、accredited investor、transfer agent。
