# lien — Architecture Decision Records

Design decisions where RWA constraints meet DeFi mechanics. These are the parts
worth discussing in a review/interview, with the trade-offs made explicit.

## ADR-1 · Liquidating permissioned collateral

**Problem.** The collateral (RWA) is a *permissioned* token: only KYC-verified
addresses may hold it. Morpho's `liquidate()` sends the seized collateral to
`msg.sender`. So a normal liquidator calling the engine directly **reverts** when
the engine tries to transfer it the seized RWA — the transfer hits the token's
compliance hook (`recipient not verified`). If liquidation can't happen, an
underwater position becomes **unrecoverable bad debt**. This is the single
hardest design point of an RWA-collateral lending market.

**Options considered.**

| Design | Mechanism | Pro | Con |
| --- | --- | --- | --- |
| **1. Liquidator allowlist** | Only KYC'd keepers may liquidate (they can receive RWA) | Zero new infra; keeper keeps the discounted RWA | Shrinks the keeper set → worse liquidation coverage, the dust-position / no-incentive risk grows |
| **2. KYC'd router + USDC buffer** *(chosen, implemented)* | A verified router receives the seized RWA, fronts the repay from a stablecoin buffer, pays the keeper the incentive in **USDC** | Liquidation stays **permissionless and capital-free** for keepers → best coverage | Router carries RWA inventory + NAV risk until it redeems (T+N); needs a funded buffer |
| **3. Protocol-custody disposal** | Seized RWA → protocol custodian, disposed later; keeper paid from a buffer | Similar coverage to (2) | More moving parts; custody centralization |

**Decision.** Implement **(2)** (`LiquidationRouter`), keep **(1)** available
(a KYC'd keeper can always call the engine directly — tested). Rationale: a
permissioned-collateral market is only as solvent as its *worst-case* liquidation
coverage; (2) maximizes the keeper set (anyone, no capital) which is exactly what
defends solvency. The cost is an inventory/NAV risk that the protocol — not the
keeper — absorbs, funded by the buffer and unwound through normal redemption.

**Residual risks (documented, not hidden).** Router NAV risk between seize and
redeem; buffer must be kept funded (a depleted buffer degrades to design 1);
redemption queue (T+N) ties up buffer capital. Tests:
`test_router_liquidation_by_non_kyc_keeper_succeeds`,
`test_router_inventory_redemption_refills_buffer`,
`test_direct_liquidation_by_non_kyc_reverts` (the problem),
`test_direct_liquidation_by_kyc_keeper_succeeds` (design 1).

## ADR-2 · NAV oracle trust model

For an RWA, the oracle **is the asset's truth** (off-chain T-bill value), so it
carries more responsibility than a DeFi price feed. Guards on `NavOracle`:

- **Staleness** — `navChecked()` reverts if the last update is older than
  `maxStaleness`. The Morpho adapter routes through it, so a halted feed
  **freezes the whole market** (borrow/withdraw/liquidate revert) rather than
  pricing positions off a dead value.
- **Circuit breaker** — `setNav` rejects a per-update move larger than
  `maxDeviationBps` (fat-finger / compromised feed). A genuine large move uses
  `forceSetNav` (admin), which is logged.

**Trade-off (the sharp edge).** "Freeze on stale" is fail-safe for *borrowing*
but it also **blocks liquidations** exactly when the feed is down — a position
could be underwater and unliquidatable. Mitigations: a generous staleness window
sized to the real NAV cadence, an oracle keeper SLA, and (future) a fallback
grace mode. We accept freeze-on-stale because pricing liquidations off a dead
feed is strictly worse. Test: `test_stale_nav_blocks_liquidation`.

## ADR-3 · Agent powers (centralization, owned honestly)

The RWA token's agent can `mint/burn/freeze/forceTransfer/recover`. This is
**deliberate** — a compliant security legally must support sanctions freezes,
court-ordered seizures, and lost-key recovery; a permissionless token cannot.
Mitigations: assign `AGENT_ROLE` to a multisig/timelock at deployment; every
privileged action emits an auditable event. This is a faithful subset of
ERC-3643 (T-REX). The point isn't to pretend it's decentralized — it's to make
the trust boundary explicit and auditable.

## ADR-4 · Decimals & rounding (correctness)

Three different scales meet in this system: **USDC 6-dec**, **RWA 18-dec**, and
**NAV 1e18**, plus Morpho's **1e36** oracle scale. Every conversion reads each
token's *real* decimals (never assumes 18) and rounds **down** (favouring the
protocol, so a round-trip at flat NAV never returns more than was put in):

- subscribe: `rwaUnits = mulDiv(usdc, 1e18·rwaScale, usdcScale·nav)`
- redeem: `usdc = mulDiv(rwa, nav·usdcScale, rwaScale·1e18)`
- Morpho adapter: `price = navWad · 10**(18 + loanDec − collDec)` = `navWad · 1e6`
  for USDC(6)/RWA(18); Morpho then does `collateralValue = collateral · price / 1e36`.

Decimal/scale mismatches are the most common lending bug class; they are pinned
with exact-value tests (`test_decimals_exact_at_par`,
`test_adapter_price_decimals_factor`) rather than left implicit.
